# Deploying introspection surfaces (`STAPEL_INTROSPECTION`)

> Frontend guardrails §5 — env-gating of introspection. This is the frontend
> mirror of the backend's `get_dev_urls()` convention (swagger / ReDoc / OpenAPI
> schema / MCP schema / debug toolbar mount only when `DJANGO_ENV ∈ {local, dev}`).

Introspection surfaces are the developer-facing self-description of a build: the
**design-system showcase** (`@stapel/showcase-viewer`, a Ladle app rendering
every package's demos) and future **report artifacts** (analytics report, G5,
generated by `stapel-tools`). Neither is a customer-facing product surface.

## Two layers of gating

**1. By construction (the primary layer).** The showcase and reports are
**separate entry points** — standalone static apps, never imported by a pair
(`@stapel/auth-react` and friends). A production bundle therefore contains
**zero introspection bytes**: gating is not "a route is turned off" but "there is
nothing to leak". This is enforced mechanically — see
`packages/auth-react/test/prodBundlePurity.test.ts` (asserts `@stapel/showcase`
is a devDependency only and `demo/` is excluded from the published tarball).
`manifest.json` / `llms.txt` shipped *inside* the npm packages are **not** gated
— they describe the public OSS artifact, not a customer's project.

**2. By deploy flag (this document).** Whether the showcase artifact is even
built and served for a given environment is decided by `STAPEL_INTROSPECTION`,
resolved by `scripts/introspection-gate.mjs`:

| Precedence | Signal | Result |
|---|---|---|
| 1 | `STAPEL_INTROSPECTION` ∈ `1\|true\|on\|yes` | **on** |
| 1 | `STAPEL_INTROSPECTION` ∈ `0\|false\|off\|no` | **off** |
| 2 | else `DJANGO_ENV` ∈ `local\|dev` | **on** |
| 2 | else `DJANGO_ENV` set to anything else | **off** |
| 3 | else (neither set) | **off** (production-safe default) |

Same signal (`DJANGO_ENV`) and same semantics as the backend `get_dev_urls()`,
so frontend and backend open/close their service surfaces in lockstep with one
deploy variable.

- `pnpm showcase:build` runs through the gate: **on** → builds + precompresses;
  **off** → prints a notice and no-ops with exit 0 (a CI "build the showcase" job
  stays green in a prod context instead of shipping the artifact). Force locally
  with `STAPEL_INTROSPECTION=1 pnpm showcase:build`.
- `pnpm introspection:gate` is the bare predicate (exit 0 = on, 1 = off) for
  composing your own steps, e.g. `pnpm introspection:gate && ./publish-reports.sh`.
- `pnpm showcase` (dev serve) is intentionally **not** gated — running the
  viewer on a workstation is local dev by definition.

### CI usage

In a workflow that publishes the showcase to a dev stand, gate the whole job on
the environment rather than branching in bash:

```yaml
- name: Build design-system showcase (introspection)
  if: env.DJANGO_ENV == 'dev' || env.DJANGO_ENV == 'local'
  run: corepack pnpm showcase:build
  env:
    STAPEL_INTROSPECTION: "1" # explicit; artifact only produced for dev stands
```

## Serving: nginx recipe (basic auth + brotli_static)

The build is a static directory (`packages/showcase-viewer/build/`), minified by
Vite and **precompressed** by the build wrapper: every text asset ≥1 KB gets a
sibling `.br` (Brotli q11) and `.gz` (gzip 9) written next to it, so nginx serves
the precompressed variant without compressing per request.

Optional dev-stand protection (frontend-guardrails §5.3): put **basic auth** in
front of all introspection paths. Mount every introspection surface under one
prefix (`/__stapel__/`) so a single `location` covers the showcase, reports, and
any proxied backend swagger paths.

```nginx
# --- Introspection surfaces (dev stands only; deployed iff STAPEL_INTROSPECTION on) ---
# brotli_static/gzip_static require the ngx_brotli module (nginx bundles
# gzip_static; brotli is the arut/ngx_brotli dynamic module). Both serve a
# precompressed sibling (asset.br / asset.gz) when present — produced by
# `pnpm showcase:build`.
server {
  # ... server_name, listen, TLS ...

  # Design-system showcase + report artifacts, behind basic auth.
  location /__stapel__/ {
    auth_basic            "Stapel introspection";
    auth_basic_user_file  /etc/nginx/.stapel_introspection.htpasswd;

    alias /srv/stapel/showcase-viewer/build/;
    index index.html;
    try_files $uri $uri/ /__stapel__/index.html;

    brotli_static on;   # serve asset.br  when the client sends Accept-Encoding: br
    gzip_static   on;   # serve asset.gz  fallback

    # Hashed asset filenames (Vite) are content-addressed → cache hard.
    location ~* /__stapel__/assets/ {
      expires 1y;
      add_header Cache-Control "public, immutable";
      brotli_static on;
      gzip_static   on;
    }
  }

  # Optional: proxy the backend's swagger/ReDoc under the same guarded prefix,
  # so one basic-auth realm covers the whole introspection surface.
  location /__stapel__/api-docs/ {
    auth_basic            "Stapel introspection";
    auth_basic_user_file  /etc/nginx/.stapel_introspection.htpasswd;
    proxy_pass            http://backend/api/schema/swagger-ui/;
  }
}
```

Create the htpasswd file with `htpasswd -c /etc/nginx/.stapel_introspection.htpasswd <user>`.

**Alternative — Django-served surfaces:** for introspection served *through*
Django (not this static bundle), reuse the existing permission-class pattern
(`IsStaffUserForSwagger`) instead of nginx basic auth. Both are options *on top
of* the `STAPEL_INTROSPECTION` deploy gate, not replacements for it.
